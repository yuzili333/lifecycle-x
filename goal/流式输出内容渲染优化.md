# 存续期数据探针智能体｜流式输出内容稳定渲染优化

你现在是一个资深 TypeScript / React / Electron / 流式渲染 / Markdown Renderer / AI Chat UI 工程师。请围绕项目 **“存续期数据探针智能体 / Cycle Data Intelligence Agent”** 优化现有的 **“流式输出内容”** 功能。

当前 `ChatComposer` 在接收大模型流式输出时存在以下问题：

1. Markdown 内容刚渲染一部分，就被中途识别为报告内容；
2. 已经显示的 Markdown 被立即清理；
3. 页面随后更新为报告卡片；
4. 由于识别、清理和重新渲染频繁发生，视图出现高频闪烁、跳动和布局抖动；
5. 模型输出的普通 `text` 内容可能被报告卡片整体覆盖或替换；
6. Markdown 是否属于报告，目前可能依赖流式片段的启发式识别，导致状态频繁反复。

本次需要建立稳定的流式内容状态机，使普通文本、普通 Markdown、报告 Markdown 和报告卡片能够独立、稳定地渲染。

请直接检查当前仓库并推进实现，不要只输出设计方案。

请优先遵守当前项目结构，复用现有 `ChatComposer`、Streaming Model Adapter、Markdown Renderer、Report Tool、Artifact Manager 和报告卡片组件，不要大规模重构无关模块。

---

## 1. 核心目标

请完成以下目标：

1. `ChatComposer` 能稳定渲染流式输出的 Markdown；
2. Markdown 流式过程中不得因“疑似报告”而被立即清空；
3. Markdown 内容必须先完整显示；
4. 流结束后进入稳定缓冲期；
5. 缓冲期结束后，只有确认属于报告的 Markdown 才转换为可点击报告卡片；
6. 报告卡片转换只允许发生一次；
7. 模型输出的普通 `text` 内容始终保留；
8. 报告卡片只能替换对应的报告 Markdown 内容段，不能替换整条 assistant 消息；
9. 普通 Markdown 内容不能被误转换为报告卡片；
10. 报告识别不能依赖未完成的 Markdown 文本片段；
11. 单个消息中同时存在 `text`、Markdown 和报告时，应分别渲染；
12. 切换报告卡片时减少布局位移、闪烁和高度突变；
13. 报告完整 Markdown 内容仍保存在 Artifact 或消息状态中；
14. 点击报告卡片后可展示完整 Markdown 报告；
15. 支持配置缓冲时间、切换动画和降级策略。

## 2. 当前问题的典型表现

当前错误流程可能类似：

```text
收到 Markdown chunk
→ 渲染部分 Markdown
→ 根据部分内容推断这是报告
→ 清空已渲染 Markdown
→ 渲染报告卡片
→ 后续 chunk 到达
→ 再次更新内容
→ 再次识别和替换
→ 页面持续闪烁和抖动
```

需要调整为：

```text
收到 text / markdown 流式事件
→ 按内容段持续追加
→ Markdown 保持稳定渲染
→ 收到明确的流结束或报告 Artifact 完成事件
→ 冻结报告 Markdown 内容
→ 进入缓冲时间
→ 一次性将报告 Markdown 节点转换为报告卡片
→ 普通 text 和普通 Markdown 保持不变
```

## 3. 核心设计原则

### 3.1 不要在流式过程中做报告卡片替换

在以下状态下禁止将 Markdown 转换为报告卡片：

* 消息流尚未结束；
* Markdown 代码块尚未闭合；
* Mermaid 尚未完整；
* 表格尚未完整；
* 数学公式尚未闭合；
* 报告 Artifact 尚未生成；
* 尚未收到明确报告完成事件；
* 仅根据标题或关键词推测为报告。

### 3.2 报告识别必须基于明确元数据

优先通过以下明确来源识别报告：

```text
report_tool_completed
report_artifact_ready
contentRole = report
contentType = markdown_report
reportArtifactId 存在
toolName = request_markdown_report_generation
```

不得仅通过以下启发式方式确定报告：

```text
内容中包含“报告”
内容以一级标题开始
Markdown 字数较长
包含多个章节
包含表格或图表
```

启发式判断最多作为低优先级 fallback，不得在流未完成时触发。

### 3.3 内容段独立管理

一条 assistant 消息可能同时包含：

```text
普通说明 text
+ 普通 Markdown
+ 报告 Markdown
+ 图表节点
+ 工具状态
```

这些内容必须拆分为稳定的内容段，报告卡片只能替换报告段。

不得使用：

```ts
message.content = reportCard;
```

替换整条消息。

应使用：

```ts
message.segments = [
  textSegment,
  markdownSegment,
  reportSegment,
  visualizationSegment
];
```

## 7. 报告转换状态机

请实现确定性的报告转换状态机。

```text
streaming
→ completed
→ buffering
→ card_ready
→ card_visible
```

异常状态：

```text
streaming/completed/buffering
→ failed
```

### 7.1 状态含义

#### streaming

* 报告 Markdown 正在接收；
* 页面正常渲染当前 Markdown；
* 禁止显示报告卡片；
* 禁止清空 Markdown。

#### completed

满足以下条件之一：

* 收到报告流完成事件；
* 收到消息流完成事件并且报告 segment 已闭合；
* 收到报告 Artifact ready 事件。

此时：

* 冻结 Markdown 内容；
* 不再接受旧 sequence 的增量；
* 保持完整 Markdown 可见。

#### buffering

* 完整 Markdown 保持显示；
* 启动缓冲计时器；
* 默认缓冲时间建议为 800～1500ms；
* 缓冲时间必须可配置；
* 缓冲阶段不得清理 Markdown；
* 如果收到新的合法增量，取消当前缓冲并回到 streaming。

#### card_ready

满足：

* 报告 Artifact 已就绪；
* 报告标题已就绪；
* Markdown 内容完整；
* 缓冲时间结束；
* 当前 segment 未被取消；
* 组件仍挂载。

#### card_visible

* 一次性显示报告卡片；
* 停止状态切换；
* 不再因后续普通 text 事件重新隐藏卡片；
* 原报告 Markdown 保留在 Artifact 或状态存储中；
* 点击卡片可打开完整报告。

## 8. 缓冲和去抖策略

请实现 `ReportTransitionController`。

### 8.1 配置

```ts
export type ReportTransitionPolicy = {
  bufferDelayMs: number;
  transitionDurationMs: number;
  requireArtifactReady: boolean;
  keepMarkdownOnTransitionError: boolean;
  allowCrossFade: boolean;

```

建议默认值：

```ts

  bufferDelayMs: 1000,
  transitionDurationMs: 180,
  requireArtifactReady: true,
  keepMarkdownOnTransitionError: true,
  allowCrossFade: true

```

### 8.2 去抖要求

* 一个报告 segment 只能创建一个有效 timer；
* 新增 delta 到达时必须取消旧 timer；
* 组件卸载时清理 timer；
* reportArtifactId 变化时重新校验；
* 相同事件不得重复触发转换；
* `card_visible` 后禁止回退，除非用户主动重新生成报告；
* 每个新报告版本使用新的 segmentId 或 reportVersion。

### 8.3 不要使用高频内容启发式判断

不要在每个 Markdown delta 到达后执行：

```ts

```

并更新整个消息状态。

报告类型应由事件元数据或工具结果确定。

### 9.1 禁止整条消息替换

```tsx

  </MessageContent>

```

### 9.2 分段渲染规则

```ts
switch (segment.type) {
  case 'text':
    return <StreamingTextSegment segment={segment} />;

  case 'markdown':
    return <StreamingMarkdownSegment segment={segment} />;

  case 'report':
    return <StreamingReportSegment segment={segment} />;

  case 'visualization':
    return <StreamingVisualizationNode segment={segment} />;

```

### 9.3 Text 内容必须始终保留


```text
textSegment:

...
```

转换后应展示：

```text
以下是本次分析的简要说明：

[风险分析报告卡片]
```

而不是只展示报告卡片。

## 10. StreamingReportSegment

请实现独立组件。

逻辑示例：

```tsx
export function StreamingReportSegment({
  segment
}: {
  segment: ReportContentSegment;
}) {
  if (
    segment.status === 'streaming' ||
    segment.status === 'completed' ||
    segment.status === 'buffering'
  ) {
    return (
      <ReportTransitionContainer mode="markdown">
        <MarkdownRenderer content={segment.markdownContent} />
      </ReportTransitionContainer>
    );
  }

  if (
    segment.status === 'card_ready' ||
    segment.status === 'card_visible'
  ) {
    return (
      <ReportTransitionContainer mode="card">
        <ReportCard
          artifactId={segment.reportArtifactId!}
          title={segment.reportTitle!}
          version={segment.reportVersion}
        />
      </ReportTransitionContainer>
    );
  }

  return (
    <MarkdownRenderer content={segment.markdownContent} />
  );
}
```

要求：

* streaming 时持续更新 Markdown；
* buffering 时 Markdown 保持可见；
* card_visible 时只转换当前报告 segment；
* 转换失败时继续展示 Markdown；
* 不显示空白内容；
* Artifact 尚未就绪时不得显示不可点击卡片。

## 11. 减少布局抖动

请实现以下措施：

### 11.1 稳定 key

内容段组件必须使用稳定的：

```text
segmentId
```

作为 React key。

禁止使用：

```text
content
array index
status
Date.now()
```

作为 key。

### 11.2 避免频繁卸载 Markdown Renderer

同一 segment 流式更新时应复用同一个组件实例。

不要在每次 delta 时：

* 修改组件类型；
* 修改 key；
* 清空节点后重新插入；
* 重建整个消息列表。

### 11.3 批量更新流式增量

对 Markdown delta 进行批处理或节流：

```ts
export type StreamRenderPolicy = {
  textFlushIntervalMs: number;
  markdownFlushIntervalMs: number;
  maxBufferedCharacters: number;
};
```

建议：

```text
text：16～32ms 批量刷新
markdown：32～80ms 批量刷新
```

不要每接收一个字符都触发完整 Markdown 解析。

### 11.4 报告切换动画

报告 Markdown 转卡片时可采用一次性淡入淡出：

```text
Markdown opacity 1 → 0
ReportCard opacity 0 → 1
```

动画持续时间建议 120～240ms。

不要使用反复折叠高度的复杂动画。

### 11.5 高度稳定策略

如果报告 Markdown 很长，直接替换为较矮卡片可能导致明显滚动跳动。

请至少实现一种稳定策略：

#### 方案 A：锚点保持

转换前记录：

* 当前滚动位置；
* 报告 segment 顶部位置；
* 用户是否处于消息底部。

转换后修正滚动位置，避免页面突然跳动。

#### 方案 B：只在用户位于消息底部时自动保持底部

如果用户正在阅读历史内容，不强制滚动到底部。

#### 方案 C：转换容器短暂保留 min-height

在切换动画期间短暂保留旧节点高度，动画完成后释放。

不得长期保留完整报告高度。

## 15. 报告识别策略

实现以下优先级：

```text
1. report_artifact_ready 事件
2. 工具调用元数据 toolName = request_markdown_report_generation
3. segment.contentRole = report
4. message metadata.reportArtifactId
5. 仅在流结束后使用有限启发式 fallback
```

即使 fallback 判断为报告，也必须满足：

* 流已结束；
* Markdown 已完整；
* Artifact 已存在或可以创建；
* 缓冲时间已结束。

## 16. 状态更新原子性

请避免以下中间状态：

```text
先清空 Markdown
→ 再设置报告卡片
```

应采用单次原子更新：

```ts
updateSegment(segmentId, {
  status: 'card_visible',
  reportArtifactId,
  reportTitle
});
```

React 渲染时直接从 Markdown 节点切换到报告卡片。

状态存储中不得出现：

```ts
markdownContent = '';
status = 'buffering';
```

Markdown 内容需要一直保留，供异常回退和完整报告查看。

```ts
    | 'markdown'
    | 'report_markdown'
    | 'visualization'
    | 'tool_status';
  delta?: string;
  sequence: number;
  metadata?: {
    contentRole?: 'general' | 'report';
    reportId?: string;
    reportArtifactId?: string;
  };
};
```

如果上游暂时只能输出单一 Markdown 流，应在报告工具调用开始时创建独立报告 segment，而不是在渲染层通过内容识别拆分。

## 21. 错误处理

定义：

```ts
export type StreamingContentErrorCode =
  | 'STREAM_SEGMENT_NOT_FOUND'
  | 'STREAM_SEQUENCE_INVALID'
  | 'STREAM_EVENT_DUPLICATED'

  | 'REPORT_TRANSITION_FAILED'
  | 'REPORT_CONTENT_EMPTY'
  | 'REPORT_METADATA_MISSING'
```

要求：

* 报告转换失败时继续展示 Markdown；
* text 内容不受错误影响；
* 单个 segment 错误不影响其他内容段；
* 不展示原始错误堆栈；
* 开发环境可记录调试日志；
* timer 和订阅必须被正确清理。
