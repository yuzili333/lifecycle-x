# Codex Goal 模式提示词：存续期数据探针智能体｜工具调用功能优化

你现在是一个资深 TypeScript / React / Electron / AI Agent / Tool Calling / Workflow Runtime 工程师。围绕项目 **“存续期数据探针智能体 / Cycle Data Intelligence Agent”** 优化和开发 **“工具调用”** 功能。

本次优化需要将以下四类能力设计为互相独立、可单独调用、可自由组合的工具方法：

1. SQL 查询；
2. Python 数据分析；
3. 绘制图表；
4. 生成报告。

用户通过自然语言提出需求后，由大模型进行意图识别，判断当前轮对话需要调用哪些工具、各工具使用什么输入，以及在存在依赖时采用什么执行顺序。

四个工具之间不设置全局强制执行顺序。每个工具都可以单独调用，也可以在同一轮或多轮会话中组合调用。系统需要在会话范围内持续保存各工具最近一次成功执行结果，并将其作为后续工具的默认输入。

请优先遵守当前仓库目录结构，不要大规模重构无关模块。

## 2. 核心目标

请实现以下能力：

1. 将 SQL 查询、Python 数据分析、绘制图表、生成报告定义为四个独立工具；
2. 用户自然语言输入后，由大模型识别本轮工具调用意图；
3. 支持一轮调用一个工具；
4. 支持一轮调用多个工具；
5. 支持工具串行执行；
6. 支持无依赖工具并行执行，当前可预留；
7. 支持同一会话中重复调用同一个工具；
8. 支持用户多轮调整参数、条件和内容要求后重新执行；
9. 保存同一会话中各工具最近一次成功执行结果；
10. 自动将最新 SQL 结果作为新发起 Python 分析或图表工具的默认输入；
11. 自动将最新 Python 分析结果作为新发起图表或报告工具的默认输入；
12. 将图表结果保存为 Artifact，作为后续报告素材；
13. 将报告保存为 Markdown Artifact；
14. 通过报告标题卡片展示报告入口；
15. 用户点击报告卡片后展示完整 Markdown 报告；
16. 保留每次工具调用的版本、来源和数据血缘；
17. 单次调用失败不覆盖上一次成功结果；
18. 不允许大模型直接伪造工具执行结果。

```text

```

```text
run

query
analysis
chart
report
```

等语义过于宽泛的名称。

## 4. 工具独立性原则

以下调用都必须支持：

### 4.1 单独调用 SQL 查询

```text
用户：查询今年以来即将到期的客户。
→ 仅调用 SQL 查询工具
```

### 4.2 单独调用 Python 分析

前提是会话中存在可用数据集：

```text
用户：分析上一轮结果中的风险评分分布。
→ 仅调用 Python 数据分析工具
```

### 4.3 单独调用绘制图表

前提是存在可用 SQL 或 Python 结果：

```text
用户：把刚才的数据画成横向柱状图。
→ 仅调用绘制图表工具
```

### 4.4 单独调用生成报告

前提是存在足够的输入 Artifact 或用户明确只需要文本报告：

```text
用户：基于刚才的分析生成报告。
→ 仅调用生成报告工具
```

### 4.5 同一轮调用多个工具

```text
用户：查询近 6 个月逾期客户，分析风险特征，画图并生成报告。
```

可生成执行计划：

```text
SQL 查询
→ Python 分析
→ 绘制图表
→ 生成报告
```

该顺序来自当前请求中的数据依赖，不是系统全局强制顺序。

### 4.6 无固定全局顺序

系统不得硬编码以下规则：

```text
SQL 必须先于 Python
Python 必须先于图表
图表必须先于报告
```

实际规则应是：

> 工具可独立执行；只有当某次调用的数据输入依赖其他工具结果时，才建立该次执行计划中的先后关系。

## 7. 会话级工具状态

请实现 `ConversationToolState`。

### 7.1 类型定义

```ts
export type ConversationToolState = {
  conversationId: string;

  latestSuccessfulSqlToolCallId?: string;
  latestSuccessfulSqlArtifactIds?: string[];

  latestSuccessfulPythonToolCallId?: string;
  latestSuccessfulPythonArtifactIds?: string[];

  latestSuccessfulChartToolCallId?: string;
  latestSuccessfulChartArtifactIds?: string[];

  latestSuccessfulReportToolCallId?: string;
  latestSuccessfulReportArtifactIds?: string[];

  selectedSqlToolCallId?: string;
  selectedPythonToolCallId?: string;
  selectedChartToolCallId?: string;
  selectedReportToolCallId?: string;

  toolCalls: ToolCallRecord[];

  updatedAt: string;
};
```

### 7.2 更新原则

只有工具成功完成后，才能更新对应的 `latestSuccessful*` 指针。

以下状态不得覆盖最近成功结果：

* failed；
* rejected；
* cancelled；
* blocked；
* waiting_approval。

例如：

```text
SQL v1 成功
SQL v2 失败
```

此时会话的最新可用 SQL 结果仍然是 SQL v1。

## 8. 默认输入解析规则

请实现 `ToolInputResolver`。

### 8.1 Python 数据分析工具

默认输入优先级：

```text
1. 用户在当前请求中明确指定的数据集或 Artifact
2. 用户在会话中手动选择的 SQL 结果
3. 会话最新一次成功 SQL 查询结果

5. 无输入时返回 waiting_input，不得伪造数据
```

### 8.2 绘制图表工具

默认输入优先级：

```text
1. 用户当前请求明确指定的 Artifact
2. 用户手动选择的 Python 分析结果
3. 会话最新一次成功 Python 分析结果
4. 用户手动选择的 SQL 查询结果
5. 会话最新一次成功 SQL 查询结果
6. 无输入时返回 waiting_input
```

### 8.3 生成报告工具

默认输入优先级：

```text
1. 用户当前请求明确指定的输入
2. 最新一次成功 Python 分析结果
3. 最新一次成功图表结果
4. 最新一次成功 SQL 查询结果
5. 当前会话中用户选择的多个 Artifact
6. 仅依据用户输入生成无数据报告模板
```

报告工具应允许同时引用：

* Python 分析摘要；
* 图表 Artifact；
* SQL 数据摘要；
* 用户补充文字；
* 既有报告版本。

### 8.4 显式指定优先

用户表达以下内容时，应覆盖默认最新结果：

```text
使用第一次查询结果
基于上一版分析
用第二张图
不要用刚才的分析
基于 SQL v2
使用报告 v1 继续修改
```

需要支持通过 `toolCallId`、`artifactId` 或版本号显式选择输入。

## 10. 工具执行计划

请实现 `ToolExecutionPlan`。

```ts
export type ToolExecutionPlan = {
  planId: string;
  conversationId: string;
  userMessageId?: string;
  steps: ToolExecutionPlanStep[];
  status:
    | 'draft'
    | 'ready'
    | 'executing'
    | 'waiting_approval'
    | 'completed'
    | 'failed'
    | 'cancelled';
  createdAt: string;
  updatedAt: string;
};
```

```ts
export type ToolExecutionPlanStep = {
  stepId: string;
  toolKind: ToolKind;
  toolName: string;
  purpose: string;
  dependencies: string[];
  inputStrategy:
    | 'explicit'
    | 'latest_sql'
    | 'latest_python'
    | 'latest_chart'
    | 'selected_artifacts'
    | 'none';
  status: ToolCallStatus;
};
```

### 10.1 执行顺序原则

* 无依赖的单工具调用直接执行；
* 有显式依赖时按拓扑顺序执行；
* 不要硬编码固定四步流水线；
* 计划中出现循环依赖时必须阻断；
* SQL 和 Python 工具继续遵守各自审批流程；
* 图表和报告是否审批由现有权限策略决定；
* 上游工具失败时，依赖该结果的下游步骤不得继续；
* 不依赖失败步骤的其他步骤可按策略继续。

## 18. 工具调用提示词约束

请提供系统提示词，用于约束大模型进行工具意图识别和调用。

```text
你可以使用四个独立工具：

1. request_sql_query_execution
2. request_python_analysis_execution
3. request_chart_rendering
4. request_markdown_report_generation

根据用户当前轮需求判断需要调用哪些工具。

四个工具可以单独调用，也可以组合调用。不要假设必须按照 SQL、Python、图表、报告的固定顺序执行。只有存在数据依赖时，才建立执行先后关系。

输入解析规则：

- Python 分析工具未显式指定输入时，默认使用当前会话最近一次成功的 SQL 查询结果。
- 绘制图表工具未显式指定输入时，优先使用最近一次成功的 Python 分析结果；如果不存在 Python 结果，则使用最近一次成功的 SQL 查询结果。
- 生成报告工具未显式指定输入时，优先使用最近一次成功的 Python 分析结果，并引用最近生成的图表 Artifact；必要时使用最近 SQL 查询摘要。
- 用户明确指定某一历史版本、toolCallId 或 artifactId 时，必须优先使用用户指定结果。

不要把工具调用计划误认为工具执行结果。不要编造 SQL 结果、Python 分析结果、图表或报告。

用户要求修改查询条件、分析规则、图表形式或报告内容时，应创建新的工具调用版本，不得覆盖已有成功版本。
```

## 19. Tool Execution Engine

请实现 `ToolExecutionEngine`。

职责：

1. 接收 ToolExecutionPlan；
2. 验证计划；
3. 解析每一步输入；
