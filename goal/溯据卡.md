# 存续期数据探针智能体｜报告“溯据卡”与分析证据链能力开发

你现在是一个资深 TypeScript / Electron / AI Agent / 数据治理 / Artifact Lineage / 审计追踪 / Markdown Report 工程师。请使用 **goal 模式**，围绕项目：

> **存续期数据探针智能体 / Cycle Data Intelligence Agent**

为现有报告系统新增核心章节：

> **溯据卡**

“溯据卡”用于展示报告结论和指标背后的可追溯证据链，体现本智能体不是替银行作风险判断的聊天机器人，而是：

> **执行、记录并证明数据分析过程的受控智能体。**

本轮应在当前报告生成、Artifact、工具执行记录和数据血缘能力基础上增量实现，不重写现有 SQL、Python、图表、报告或工作流模块。

请直接检查当前仓库并推进实现，不要只输出设计方案。

优先遵守当前项目结构、类型定义、Artifact 协议、工具调用记录、报告模板和 Astryx UI 规范，不要大规模重构无关模块。

## 1. 当前背景

项目当前已经支持：

1. SQL 数据查询；
2. Python 数据分析；
3. 图表生成；
4. Markdown 报告生成；
5. SQL、Python、图表和报告 Artifact；
6. 工具执行记录；
7. 上下游 Artifact 引用；
8. 部分数据血缘；
9. 完整报告中的真实图表渲染；
10. 报告卡片和完整报告查看页面。

当前报告主要展示：

* 分析结果；
* 指标；
* 图表；
* 文字结论；
* 风险提示或管理建议。

但报告尚不能系统回答：

1. 数据来自哪里；
2. 分析覆盖了哪些数据；
3. 使用了哪些筛选条件；
4. 指标通过什么公式计算；
5. 实际执行了哪些 SQL；
6. 实际执行了哪些 Python 分析；
7. 当前结论引用了哪些 Artifact；
8. Artifact 之间有什么上下游关系；
9. 报告结果能否追溯至原始数据和工具执行；
10. 哪些内容是数据事实，哪些内容是模型生成的解释。

需要通过“溯据卡”补全这一能力。

---

## 2. “溯据卡”的定位

“溯据卡”不是普通的数据来源说明，也不是技术日志原样输出。

它是报告中用于证明分析过程的结构化证据章节。

溯据卡应至少覆盖：

* 数据来源；
* 分析范围；
* 筛选条件；
* 统计公式；
* SQL 执行记录；
* Python 执行记录；
* 上下游 Artifact；
* 数据血缘。

其核心声明是：

> 本报告中的数据指标和图表来源于受控工具的实际执行结果。智能体负责组织分析流程、调用工具、记录证据和生成说明，不替代银行既有风险分类制度、审批制度或人工风险判断。

## 4. 核心设计原则

### 4.1 证据来自系统记录，不来自模型推测

以下内容必须由程序从真实记录构建：

* 数据源；
* 表；
  -字段；
* 筛选条件；
* SQL；
* Python 执行；
* Artifact ID；
* 工具调用时间；
* 工具状态；
* 数据血缘；
* 输入输出关系；
* 数据行数；
* 数据集摘要。

模型只能：

* 将结构化证据组织为可读说明；
* 对统计公式进行格式化；
* 生成简短的分析方法描述；
* 解释数据限制。

模型不得：

* 编造不存在的 SQL；
* 编造未执行的 Python；
* 编造 Artifact；
* 推断数据源；
* 将失败工具调用描述为成功；
* 将样例数据描述为全量数据；
* 将模型文字解释标记为数据证据。

---

### 4.2 数据事实与判断分离

报告应清楚区分：

```text
数据事实
→ 工具实际执行产生的数值、表格和图表

统计解释
→ 基于真实结果的描述性说明

风险判断
→ 银行风险制度和人工审批范围
```

推荐在溯据卡中增加声明：

```text
本溯据卡证明数据分析过程及结果来源，不构成授信审批、风险分类调整或风险处置决定。
```

---

### 4.3 报告正文与证据链分离但互相引用

正文负责：

* 结论；
* 图表；
* 指标；
* 建议；
* 限制说明。

溯据卡负责：

* 数据证据；
* 执行证据；
* 计算证据；
* 血缘证据。

正文中的关键指标可引用：

```text
证据项 E-003
```

但 P0 不要求为每句话增加脚注。

P0 至少要求报告级溯据卡能够覆盖全部主要结果。

## 17. 证据完整性校验


8. 是否错误引用失败工具结果；
9. 是否引用过期或已删除 Artifact。

## 18. 缺失证据处理

如果部分证据缺失：

```ts
status: 'partial'
```

报告中显示：

```text
当前溯据链不完整：缺少统计公式与实际执行表达式之间的验证记录。
```

禁止显示：

```text
全部分析过程已验证
```

如果核心数据来源或 SQL 结果不存在：

```ts
status: 'invalid'
```

不应生成带具体数据结论的正式报告。

可进入工作流异常恢复，引导用户：

* 重新执行查询；
* 重新执行分析；
* 恢复失效 Artifact；
* 选择有效数据源。

## 19. 报告中的章节位置

“溯据卡”应作为报告核心章节，不是附件中一句话。

推荐结构：

```markdown
# 报告标题

## 一、执行摘要

## 二、分析范围

## 三、核心结果

## 四、可视化分析

## 五、分析结论与提示

## 六、溯据卡

## 七、数据限制与使用边界
```

对于已有报告模板，建议将“溯据卡”放置在：

```text
分析结论之后
数据限制之前
```

如现有模板结构不同，保持其主结构，仅新增一个固定章节。

## 31. 数据准确性约束

必须明确：

```ts
export type EvidenceAccuracyPolicy = {
  allowFabricatedEvidence: false;
  allowSyntheticExecutionRecord: false;
  allowMissingArtifactGuessing: false;
  allowFailedToolAsSuccessfulEvidence: false;
  requireSqlOrPythonForNumericEvidence: true;
  requireLineageForVisualizationEvidence: true;
};
```

以下情况不得生成“证据完整”的报告：

* 没有成功 SQL；
* 数值没有真实 Artifact；
* 图表没有上游数据来源；
* 工具记录缺失；
* 公式无法关联至执行逻辑；
* Artifact 已过期且无法解析。

## 33. 错误和降级

```ts
export type EvidenceCardErrorCode =
  | 'EVIDENCE_CARD_NOT_FOUND'
  | 'REPORT_ARTIFACT_NOT_FOUND'
  | 'DATA_SOURCE_EVIDENCE_MISSING'
  | 'TOOL_EXECUTION_RECORD_MISSING'
  | 'FORMULA_EVIDENCE_MISSING'
  | 'ARTIFACT_LINEAGE_INCOMPLETE'
  | 'ARTIFACT_EXPIRED'
  | 'EVIDENCE_PERMISSION_DENIED'
  | 'EVIDENCE_VALIDATION_FAILED'
  | 'UNKNOWN_ERROR';
```

降级规则：

* 某一项缺失：显示“部分完整”；
* 工具详情不可用：保留其他证据；
* 血缘图失败：降级为文本链路；
* SQL 详情加载失败：显示执行概要；
* EvidenceCard 整体失败：报告正文仍可打开，但明确显示证据不可用；
* 不得把异常静默隐藏。
