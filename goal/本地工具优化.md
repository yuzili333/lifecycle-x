# 存续期数据探针智能体｜复合工具规划、SQL 输入补全与图表报告融合优化

你现在是一个资深 TypeScript / Electron / AI Agent / Tool Calling / Workflow Runtime / Prompt Engineering 工程师。围绕项目 **“存续期数据探针智能体 / Cycle Data Intelligence Agent”** 优化 Agent 的工具规划、工具依赖补全、图表调用和报告内容生成能力。

本轮只完成以下范围：

1. 同时支持复合需求和单工具需求；
2. 在缺少 SQL 查询结果时，为 Python、图表、报告工具自动补充 SQL 查询步骤；
3. 用户明确要求绘制图表时，必须稳定命中图表工具；
4. 同时要求图表和报告时，将图表结果嵌入报告正文；
5. 精简 Agent 系统提示词、工具提示词和预制对话提示词，降低模型推理耗时；
6. 优先遵守当前项目结构，不大规模重构无关模块。

请直接检查当前仓库并推进实现，不要只输出设计方案。

## 1. 当前问题

目前存在以下问题：

### 1.1 复合需求推理耗时过长

用户输入可能同时包含：

```text
查询数据
分析数据
绘制图表
生成报告
```

现有 Agent 可能：

* 多次重新识别意图；
* 多次重新构造上下文；
* 每执行一个工具后再次调用模型规划；
* 重复分析工具顺序；
* 重复判断字段和数据源；
* 将大量无关工具说明和历史上下文传入模型；
* 使用长篇自然语言描述工具编排过程。

导致复合需求平均推理耗时超过 30 秒。

### 1.2 图表工具命中率低

即使用户明确输入：

```text
绘制图表
生成柱状图
可视化展示
画出分布图
```

Agent 仍可能：

* 只执行 SQL 查询；
* 只生成 Markdown 表格；
* 直接生成报告；
* 由模型在正文中描述图表，而没有调用图表工具。

### 1.3 图表与报告内容分离

当前同时执行：

```text
绘制图表
生成报告
```

后，图表可能作为独立消息或独立 Artifact 展示，报告正文中没有图表节点，导致报告内容与数据可视化脱节。

## 2. 工具范围

本轮只围绕以下四个工具优化：

```text
request_sql_query_execution
request_python_analysis_execution
request_chart_rendering
request_markdown_report_generation
```

四个工具保持独立注册。

不得将四个工具合并成一个不可拆分的复合工具。

不得为所有请求硬编码固定执行顺序。

执行顺序应由：

* 用户明确要求；
* 工具输入依赖；
* 已存在的成功结果；
* 当前工作流状态；

共同决定。

## 3. 核心设计原则

### 3.1 用户目标优先

首先识别用户明确要求的输出：

```text
查询结果
分析结果
图表
报告
```

用户明确要求的输出不得被规划器遗漏。

### 3.2 图表是强约束输出

当用户明确包含以下意图时：

```text
绘制图表
画图
生成图表
可视化
柱状图
折线图
饼图
分布图
趋势图
结构图
```

必须将：

```text
request_chart_rendering
```

加入执行计划。

不得由以下内容替代图表工具：

* Markdown 表格；
* Mermaid；
* ASCII 图；
* 纯文字图表描述；
* 模型自行输出的图表配置；
* 报告中的文字总结。

### 3.3 SQL 结果是数据工具链的基础输入

Python 分析、图表绘制和数据报告生成必须能够追溯到真实 SQL 查询结果。

满足以下任一条件时，视为具备 SQL 输入：

1. 当前轮存在成功 SQL 结果；
2. 历史对话存在可复用的成功 SQL 结果；
3. 当前 Python、图表或报告 Artifact 的数据血缘中包含成功 SQL 结果；
4. 用户明确指定一个由 SQL 结果生成的数据集 Artifact。

如果不满足，应自动补充 SQL 查询步骤。

### 3.4 不固定工具顺序

以下流程都应支持：

```text
SQL
```

```text
SQL → Python
```

```text
SQL → 图表
```

```text
SQL → 报告
```

```text
SQL → Python → 图表
```

```text
SQL → 图表 → 报告
```

```text
SQL → Python → 报告
```

```text
SQL → Python → 图表 → 报告
```

如果已有可复用 SQL 结果，也应支持：

```text
已有 SQL 结果 → Python
已有 SQL 结果 → 图表
已有 SQL 结果 → 报告
已有 SQL 结果 → Python → 图表 → 报告
```

顺序只能由依赖关系决定，不得在系统提示词中强制所有请求执行四个工具。

## 4. 支持的需求类型

### 4.1 单工具需求

用户只要求执行一个工具目标。

示例：

```text
基于刚才的查询结果绘制柱状图。
```

计划：

```text
request_chart_rendering
```

前提：

* 能找到可用的 SQL 查询结果；
* 或找到具有 SQL 数据血缘的分析 Artifact。

示例：

```text
把刚才的数据生成报告。
```

计划：

```text
request_markdown_report_generation
```

如果报告需要图表但用户未明确要求图表，不得擅自增加图表工具。

### 4.2 复合需求

用户在一轮中明确要求多个结果。

示例：

```text
查询风险分类数据，分析笔数和金额分布，绘制图表并生成报告。
```

计划至少包含：

```text
request_sql_query_execution
request_python_analysis_execution
request_chart_rendering
request_markdown_report_generation
```

顺序由输入依赖计算。

### 4.3 已有结果上的复合需求

示例：

```text
基于上一轮查询结果继续分析，绘制图表并生成报告。
```

如果上一轮 SQL 结果仍有效：

```text
request_python_analysis_execution
request_chart_rendering
request_markdown_report_generation
```

不得重复执行 SQL。

## 5. 意图识别输出模型

请优化 Agent 的意图识别输出，使其结构化、简短且可直接用于工具规划。

```ts
export type AgentToolIntent = {
  requestType:
    | 'single_tool'
    | 'compound';

  goals: {
    query: boolean;
    analysis: boolean;
    chart: boolean;
    report: boolean;
  };

  explicitGoals: {
    query: boolean;
    analysis: boolean;
    chart: boolean;
    report: boolean;
  };

  dataRequirements: {
    requiresSqlResult: boolean;
    canReuseExistingSqlResult: boolean;
  };

  analysisGoal?: string;
  chartGoal?: string;
  reportGoal?: string;

  unresolvedInputs: string[];
};
```

### 5.1 explicitGoals

`explicitGoals` 只记录用户明确表达的目标。

例如：

```text
绘制图表并生成报告
```

必须得到：

```json
{
  "explicitGoals": {
    "query": false,
    "analysis": false,
    "chart": true,
    "report": true
  }
}
```

后续规划器不得删除显式目标。

### 5.2 图表意图后置校验

在模型返回意图结果后，增加确定性校验：

```ts
export function enforceExplicitOutputGoals(
  userMessage: string,
  intent: AgentToolIntent
): AgentToolIntent;
```

如果用户文本明确包含图表意图，而模型返回：

```ts
goals.chart === false
```

程序应将其修正为：

```ts
goals.chart = true;
explicitGoals.chart = true;
```

该校验只用于明确图表表达，不应对模糊文本擅自增加工具。

## 7. SQL 输入解析优先级

```ts
export type SqlInputResolution = {
  status:
    | 'resolved'
    | 'requires_sql_step'
    | 'requires_user_input';

  source:
    | 'explicit_artifact'
    | 'current_round'

  datasetArtifactId?: string;

  fallbackQueryRequest?: SqlFallbackQueryRequest;

```

解析优先级：

```text
1. 用户显式指定的 SQL 结果或数据集 Artifact
2. 当前轮最新成功 SQL 结果
```

以下状态不得被复用：


## 8. 自动补充 SQL 查询步骤

当用户直接要求：

```text
Python分析
绘制图表
生成报告
```

但不存在可复用 SQL 结果时，规划器应自动检查是否可以补充 SQL 查询。

### 8.1 已选择可用数据源

如果当前存在：

* 已选择数据源；
* 明确的活动表；
* 用户具有查询权限；
* 数据源状态可用；

则自动添加：

```text
request_sql_query_execution
```

作为前置步骤。

例如用户输入：

```text
绘制这个数据的分布图。
```

当前已选择一个 CSV 临时表，但没有历史 SQL 结果。

计划：

```text
SQL 全量查询
→ 图表
```

### 8.2 全量数据兜底的准确含义

“全量数据”是指：

* 不添加业务筛选条件；
* 不使用样例数据；
* 不使用随机抽样；
* 不使用 preview rows；
* 查询当前已选择活动表的完整数据范围。

如果能够从用户目标识别必要字段，可只查询必要字段，但必须包含全部数据行。

例如：

```sql
SELECT
  "五级分类",
  "贷款余额"
FROM "chat_csv_xxx";
```

如果无法识别必要字段，可使用：

```sql
SELECT *
FROM "chat_csv_xxx";
```

不得为了降低数据量而擅自添加：

```sql
LIMIT 100
```

除非用户明确要求抽样或预览。

### 8.3 大数据执行保护

全量查询仍需遵守现有工程保护：

* 只读 SQL；
* SQL 安全校验；
* 用户审批；
* 权限校验；
* 执行超时；
* 取消；
* 流式读取；
* 分批物化；
* SQLite 临时结果表；
* 磁盘容量检查；
* 查询熔断；
* 审计。

“全量查询”不代表把完整结果直接放入模型上下文。

SQL 结果应：

```text
查询
→ 物化为数据集 Artifact 或 SQLite 临时表
→ 下游工具引用 Artifact
```

### 8.4 多表歧义

如果数据源包含多个表，且没有活动表或无法判断目标表：

不得任意选择一个表。

应返回：

```ts
{
  status: 'requires_user_input',
  issue: {
    code: 'ACTIVE_TABLE_REQUIRED',
    message: '当前数据源包含多个数据表，请选择需要分析的数据表。'
  }
}
```

将该异常输入给 Agent，由 Agent 生成简短、具体的用户引导。

## 9. 无可用数据源时的处理

如果：

* 没有 SQL 结果；
* 没有可复用 Artifact；
* 没有已选择的数据源；

则不得：

* 使用模拟数据；
* 使用模板数据；
* 由模型编造查询结果；
* 直接生成带数值的图表或报告；
* 擅自结束工作流。

应返回结构化异常：

```ts
export type SqlInputResolutionIssue = {
  code:
    | 'SQL_RESULT_NOT_FOUND'
    | 'DATA_SOURCE_NOT_SELECTED'
    | 'ACTIVE_TABLE_REQUIRED'
    | 'DATA_SOURCE_UNAVAILABLE'
    | 'SQL_RESULT_EXPIRED';

  message: string;
  recoverable: true;
  suggestedAction:
    | 'select_data_source'
    | 'select_table'
    | 'rerun_query'
    | 'upload_csv';
};
```

Agent 接收到异常后应提示：

```text
当前没有可用于绘制图表的查询结果，也没有已选择的数据源。

请先选择一个已连接数据库、已导入 CSV，或上传本地 CSV 文件。
```

## 10. 依赖规则

### 10.1 Python 分析

Python 输入优先级：

```text
1. 用户指定数据集 Artifact
2. 当前轮 SQL 结果
3. 历史可复用 SQL 结果
4. 自动补充 SQL 查询结果
```

Python 不得直接连接业务数据库。

### 10.2 图表绘制

图表输入优先级：

```text
1. 用户指定 Python 分析 Artifact
2. 当前轮最新 Python 结果
3. 用户指定 SQL 结果 Artifact
4. 当前轮最新 SQL 结果
5. 历史可复用 SQL 结果
6. 自动补充 SQL 查询结果
```

如果图表需要分析后指标，则依赖 Python。

如果只是简单分组、计数、汇总结果，可以直接使用 SQL Artifact。

### 10.3 报告生成

报告输入必须能够追溯到 SQL 数据血缘。

报告可引用：

* SQL 查询结果；
* Python 分析结果；
* 图表 Artifact；
* 数据质量信息；
* 用户明确的报告要求。

## 11. 图表工具强制命中策略

请实现双层策略。

### 11.1 提示词层

系统提示词明确：

```text
用户明确要求“绘制图表、生成图表、可视化、柱状图、折线图、饼图、分布图、趋势图”等输出时，必须调用 request_chart_rendering。

Markdown 表格、文字描述、Mermaid 或报告正文不能替代图表工具。
```

### 11.2 程序校验层

在生成 `ToolExecutionPlan` 后执行：

```ts
export function validateExplicitToolCoverage(input: {
```

```ts
if (
  intent.explicitGoals.chart === true &&
```

同理：

* 明确要求报告时必须包含报告工具；

* 明确要求查询时必须包含 SQL 工具或复用现有 SQL 结果的明确说明。

## 12. 图表与报告融合

当用户同时要求：

```text
绘制图表
生成报告
```

执行计划中：

```text
request_markdown_report_generation
```

必须依赖：

```text
request_chart_rendering
```

### 12.1 报告工具输入

```ts
export type MarkdownReportGenerationInput = {
  reportGoal: string;

  sqlArtifactIds: string[];
  analysisArtifactIds: string[];
  visualizationArtifactIds: string[];

  includeVisualizations: boolean;

  dataSourceSummary?: string;
  requestedSections?: string[];
};
```

当用户明确要求图表和报告时：

```ts
includeVisualizations = true;
visualizationArtifactIds.length > 0;
```

### 12.2 报告 Markdown 节点

报告正文中使用统一可视化节点。

例如：

```markdown
## 三、风险分类分布

根据查询结果，正常类占比最高，关注类和不良类需重点监控。

<visualization
  artifactId="artifact_chart_001"
  title="五级分类笔数分布"
/>

<visualization
  artifactId="artifact_chart_002"
  title="五级分类贷款余额分布"
/>
```

应复用项目现有 Markdown 自定义节点、Artifact 引用或 VisualizationSpec 协议。

不得：

* 将完整 ECharts option 写入 Markdown；
* 将图片 Base64 写入报告；
* 只在报告外展示图表；
* 在报告中重新生成另一份无血缘图表。

### 12.3 避免重复展示

当用户同时要求图表和报告时，默认：

* 图表 Artifact 由图表工具创建；
* 报告正文引用该 Artifact；
* ChatComposer 最终以报告卡片为主要入口；
* 不额外在报告卡片外重复展示相同图表，除非当前产品规范明确要求或用户要求单独展示。

### 12.4 图表失败处理

如果用户明确要求图表和报告，而图表生成失败：

不得静默生成一个不包含图表的报告并宣称任务完成。

应：

1. 保留 SQL 和 Python 成功结果；
2. 尝试修复或重试图表；
3. 如需用户补充字段或图表类型，进入可恢复状态；
4. 用户明确同意无图表报告后，才允许跳过图表。

## 14. Agent 系统提示词优化

请精简现有 Agent 系统提示词，删除重复、含糊和无执行价值的内容。

推荐核心系统提示词：

```text
你是 Cycle Probe 数据分析智能体。你的任务是理解用户希望获得的查询、分析、图表和报告结果，并生成最小且完整的工具执行计划。

可用工具：
- request_sql_query_execution：查询、筛选、分组、聚合和排序数据。
- request_python_analysis_execution：对 SQL 结果执行统计、趋势、异常或其他分析。
- request_chart_rendering：将 SQL 或 Python 结果绘制为可交互图表。
- request_markdown_report_generation：基于真实 SQL、分析和图表结果生成 Markdown 报告。

规划规则：

1. 用户可以只要求一个工具结果，也可以同时要求多个结果。
2. 不固定工具顺序，根据数据依赖生成执行计划。
3. 用户明确要求绘制图表或可视化时，必须调用 request_chart_rendering。
4. 用户明确要求报告时，必须调用 request_markdown_report_generation。
5. SQL 结果是 Python、图表和报告的数据基础。
```

* 用户修改了工作流目标；
* 工具结果改变了原计划的可执行性。

普通成功结果应由工作流引擎按照既定计划继续执行。

## 26. 异常恢复

### 26.1 SQL 自动补充失败

例如：

* 多表歧义；
* 数据源不可用；
* 权限不足；
* 临时数据源过期。

Agent 只生成简短用户引导，不重新执行整套意图规划。

例如用户要求柱状图，但数据集中只有文本字段。

应：

* 保留 SQL 结果；
* 提示选择数值字段；

* 不使用模拟值。

### 26.3 报告缺少图表

如果显式目标包含图表和报告，而图表 Artifact 不存在：

* 报告步骤保持 `waiting_input`；
* 修复或重新执行图表；
* 不静默跳过图表。
